# Architecture

oh-my-claude is a harness that wraps Claude Code's lifecycle events with bash hooks. Every prompt, tool use, compaction, resume, and stop attempt passes through scripts that classify intent, inject context, track state, and enforce quality gates. There is no daemon, no runtime, and no external service. The system is bash scripts sourcing a shared library, communicating through a primary JSON state file plus bounded JSONL/log and transient per-tool sidecars.

---

## Components

### Orchestration

**`skills/autowork/SKILL.md`** -- Master skill definition. Entry point for `/ulw`, `/autowork`, `/ultrawork`, and `/sisyphus` commands. Defines thinking requirements (plan before acting, reflect after results, no mechanical tool chaining), operating rules (classify intent, classify domain, route to specialists), and execution style (incremental changes, mandatory review, no premature stopping). Sets `disable-model-invocation: true`; carries no `model:` pin (v1.49) so the skill runs on the current session model without imposing an Opus ceiling.

**`skills/council/SKILL.md`** -- Adaptive multi-specialist evaluation. Council derives a coverage map from the task and repository evidence, considers the full available agent roster (bundled and custom), and records relevant include/skip reasons. It dispatches the smallest sufficient primary team concurrently (normally 1–4; one is valid for a narrow specialist question), waits for every primary return, reconciles tensions and overlaps, then may run one optional 0–2-agent gap-fill round for a concrete uncovered question. Four is a normal envelope rather than a hard ceiling: a larger primary is permitted only when the ledger records the cost and every mutually independent, high-impact competence need. `record-council-coverage.sh` enforces primary-only initialization, immutable dispatched objectives/rows, exact-objective returns, post-primary reconciliation, the one-round gap-fill cap, and final completion; `primary-complete` requires every selected result evidenced, while explicit residual limitations may remain after `gap-fill-complete`. `record-pending-agent.sh` rejects tagged dispatches absent from the current lifecycle. Zero to three load-bearing findings may receive an independent, competence-matched verification pass, authorized only after final reconciliation; verifier identities are unique, provenance-stamped, and completion waits for every dispatched verifier. `--polish` and `--self-audit` add coverage priors rather than fixed rosters. `--deep` passes Opus only to selected Sonnet-backed agents and omits an explicit model for `model: inherit` agents, preserving stronger session models. A narrow explicit-uncertainty signal activates the same deep route without requiring the user to know the flag and makes one best-fit inherited deliberator part of the smallest sufficient set; generic breadth/high risk does not. Only a completed assessment leaves a revision-bound Phase 8 handoff; the immediate matching implementation follow-up reuses it rather than recreating a panel. Phase 8 entry and exhaustive scope authorization remain separate predicates.

### Shared Library

**`skills/autowork/scripts/common.sh`** (~11,700 lines, with five subsystems extracted to `lib/`) -- Most hook scripts source this file. State I/O (`lib/state-io.sh`, v1.12.0) and verification scoring (`lib/verification.sh`, v1.14.0) are sourced eagerly. Prompt classification (`lib/classifier.sh`, v1.13.0) and timing capture / the `/ulw-time` card (`lib/timing.sh`) are exposed through idempotent common loaders so selected hot-path callers can defer them. Model-drift confabulation-canary computations live in `lib/canary.sh` (v1.26.0), which `canary-claim-audit.sh` sources directly rather than through `common.sh`. Together they provide:

- **JSON state management** (`lib/state-io.sh`): `write_state(key, value)`, `read_state(key)`, `write_state_batch(k1, v1, k2, v2, ...)` for atomic multi-key updates, plus `with_state_lock` / `with_state_lock_batch` for concurrent-safe read-modify-write. All operations use jq with atomic temp-file-then-mv writes to prevent corruption.
- **Verification scoring** (`lib/verification.sh`): `score_verification_confidence(cmd, output, project_test_cmd)` for Bash invocations, `score_mcp_verification_confidence(verify_type, output, has_ui_context)` for MCP tools, plus `detect_verification_method`, `classify_mcp_verification_tool`, and `detect_mcp_verification_outcome`. Pure functions over the command text and output — no state I/O.
- **Intent classification** (`lib/classifier.sh`): `classify_task_intent(text)` -- returns one of 5 categories (see classification order below). Delegates to `is_continuation_request`, `is_checkpoint_request`, `is_session_management_request`, `is_imperative_request`, and `is_advisory_request`.
- **Domain scoring** (`lib/classifier.sh`): `infer_domain(text)` -- scores prompt text against keyword lists for 6 domains (coding, writing, research, operations, mixed, general). Uses `count_keyword_matches` with grep to count occurrences. Highest score wins. `mixed` fires in two regimes: coding involvement with a second domain scoring at least 40% of the primary, or higher-confidence non-coding pairs where both domains score at least 2 and the secondary still reaches the 40% ratio.
- **Council evaluation detection**: `is_council_evaluation_request(text)` -- detects broad whole-project evaluation requests that benefit from adaptive multi-specialist coverage. Pattern families and narrow-scope guards prevent a focused feature, capability, surface, or subsystem from inflating into a whole-project council.
- **Prompt normalization**: `normalize_task_prompt(text)` -- strips `/ulw`, `autowork`, `ultrawork`, `sisyphus`, and `ultrathink` prefixes.
- **Continuation detection**: `is_continuation_request(text)` -- matches "continue", "resume", "carry on", "keep going", "pick it back up", etc.
- **Helpers**: `truncate_chars(limit, text)`, `trim_whitespace(text)`, `now_epoch`, `is_internal_claude_path(path)`, `is_maintenance_prompt(text)`, `has_unfinished_session_handoff(text)`, `is_execution_intent_value(intent)`.

### Hook Scripts

**`quality-pack/scripts/prompt-intent-router.sh`** (~2,700 lines) -- **UserPromptSubmit hook**. The master context injector. For every prompt:

1. Reads session ID and prompt text from hook JSON.
2. Classifies intent via `classify_task_intent`.
3. Resets per-turn counters (`stop_guard_blocks`, `session_handoff_blocks`, `advisory_guard_blocks`, `stall_counter`).
4. Preserves or updates `current_objective` depending on intent type. A fresh execution objective snapshots the current `edited_files.log` offset, Bash-event count, plan revision, findings-ledger signature, prompt timestamp, and strict broad-scope signal used by adaptive review coverage; continuations preserve them.
5. Appends to `recent_prompts.jsonl` (capped at 12 entries).
6. If the prompt contains an activation keyword (`ultrawork`, `ulw`, `autowork`, `sisyphus`), builds a registered `context_parts` array. Every directive is explicitly mandatory, core, or optional; unknown names fall back to lowest-priority optional with anomaly telemetry rather than receiving an unlimited exemption. A whole-payload ceiling applies after mandatory recovery/gate/user-delta entries are selected. Full routing bodies are edge-triggered on objective/domain/intent/model-tier/risk changes, compact/resume, and a 30-minute TTL; stable turns receive a compact routing-state delta.
   - Mode directive (continuation, advisory, session-management, checkpoint, or new execution).
   - Preserved objective and prior assistant state for continuations.
   - Recent specialist conclusions from `subagent_summaries.jsonl`.
   - Thinking directive (plan/reflect requirements).
   - Domain-specific specialist hints (coding, writing, research, operations, mixed, general).
   - Adaptive Council guidance (for broad whole-project evaluation requests detected by `is_council_evaluation_request`): persisted coverage map, full roster, normally 1–4 primary agents with an evidenced exception path, optional 0–2 gap-fill, competence-matched verification, and a separate Phase 8 authorization decision.
7. If the prompt contains `ultrathink`, appends a deeper investigation directive.
8. Emits the assembled context via `hookSpecificOutput.additionalContext`.

Fresh execution routing also arms the two-phase closeout protocol and clears any prior seal, display-suppression marker, finalizer claim, bounded provisional-summary sidecar, and prior generation-scoped material nonces. Continuation prompts preserve the active work generation; subsequent evidence mutations make its fingerprint stale automatically.

**`skills/autowork/scripts/closeout-preflight.sh`** -- **PostToolBatch hook plus manual fallback**. For material ULW execution, likely terminal evidence batches (reviewer returns, verification/delivery Bash, or MCP verification surfaces) trigger a readiness evaluation before the next model response. The script copies the active session into an isolated temporary `HOME`/`STATE_ROOT`, runs the real `stop-guard.sh` there with a synthetic structured closeout, and accepts only a shadow `session_outcome=completed` plus zero-or-one valid guard response. Gate events, skip/outcome state, substantive review/verification/objective/scope/dimension/goal caps, and the ULW sentinel remain isolated. Only the monotonic acknowledgements for the one-shot `wave_shape_blocks` and `shortcut_ratio_blocks` warnings can max-merge back under a full-generation CAS, because the model has received those warnings privately; repeated reads and ordinary work cannot consume any cap. An optimistic work/evidence fingerprint before evaluation, under-lock seal commit, and post-commit recheck prevent a concurrent edit/reviewer/plan/ledger write from certifying the wrong generation. Every PostToolBatch first writes an atomic nonce scoped to its captured enforcement generation, then rechecks that generation under the session lock; stale callbacks can neither publish into a newer interval nor overwrite its seal invalidation. READY injects one hidden cumulative evidence manifest; NOT READY injects the exact private recovery context. The manual command prints only a compact READY/NOT_READY receipt and refuses recovered/incomplete authority state.

**`skills/autowork/scripts/closeout-display.sh`** -- **MessageDisplay hook**. MessageDisplay fires once per whole-line streamed batch and its replacement output is bounded, so the hook deliberately does not buffer text across batches. Before READY, ordinary prose passes untouched; a neutral first batch may arm a message-ID watch, and the first later completion-shaped batch becomes one compact “checking closeout gates” marker while the rest of that same provisional message is suppressed. Once a current seal exists, every batch passes through byte-for-byte and the live Stop guard owns exact final validation. A hook timeout therefore falls open only for the current original batch and can never strand earlier accepted-final bytes in a private replay buffer. Fresh prompts clear the small message-ID/timestamp bookkeeping.

**`skills/autowork/scripts/stop-dispatch.sh`** -- the **single managed Stop hook and stdout owner**. Claude Code runs matching handlers in parallel, so settings do not rely on Stop-array order. The dispatcher reads stdin once and executes guard → timing → canary → transcript archive. A blocked guard checkpoints token/economics data only, privately retains a redacted bounded completion candidate for fact preservation, and emits one compact continuation. Claude Code 2.1.163+ receives non-error Stop `additionalContext`; legacy/unparseable clients use `decision:block`. Before shared code loads, exact per-session authority fails closed through a dependency-light marker path if `jq` is missing/broken or state is corrupt/recovered; the global marker alone never captures an ordinary Stop. The dispatcher reads `CLAUDE_CODE_STOP_HOOK_BLOCK_CAP` (default 8) and ends explicitly at `max(1, cap-1)`—the seventh continuation under the default—with an unresolved-gate receipt and bounded uncertified candidate, never a silent platform termination. Accepted ULW releases run best-effort canary/archive finalizers once and receive the visible closeout receipt. Exhausted scorecard releases replay a held candidate, while skip/no-work releases are not duplicated or mislabeled as fully passed.

**`skills/autowork/scripts/stop-guard.sh`** -- hard quality evaluator invoked by `stop-dispatch.sh` (and directly by isolated preflight/tests). It can continue Claude instead of accepting Stop. Principal checks:

1. **Advisory inspection gate**: If the task is advisory over a codebase (coding or mixed domain) and no code inspection (`last_advisory_verify_ts`) or build/test verification (`last_verify_ts`) was detected, blocks the stop. Cap: 1 block.
2. **Session handoff gate**: If the last assistant message contains deferral language ("ready for a new session", "next wave", "next phase") and the user did not request a checkpoint, blocks the stop. Cap: 2 blocks.
3. **Review/verification gate**: If files were edited (`last_code_edit_ts` / `last_doc_edit_ts` set, or legacy `last_edit_ts` for resumed sessions) but review (`last_review_ts` for code, `last_doc_review_ts` for docs) or verification (`last_verify_ts`) are missing or stale, blocks the stop. Doc-only edits route to `editor-critic` instead of `quality-reviewer` and skip the verification requirement. Verification is only checked for coding and mixed domains. Cap: 3 blocks (the `· block N` counter is exposed in the message; W2 verified 0 of 418 sessions ever reached `block_count ≥ 2` for THIS gate specifically — other gates do hit block 2 in telemetry, e.g. `pretool-intent` and `discovered-scope` — so the cap is preserved for safety). Block 1 emits a verbose message framing the reviewer's lens as "correctness AND completeness" (v1.40.x rewrite of the prior "FIRST self-assess" runbook); blocks 2+ use a concise "Still missing: X. Next: Y." form that names only un-ticked items.
4. **Adaptive dimension gate**: After the standard review/verify gate passes, the current objective's edited surfaces and semantic breadth determine extra review coverage. On each block, the gate names the missing dimension and its best owner rather than replaying a standing reviewer chain.

    - **Scope**: fresh execution prompts snapshot the current `edited_files.log` line offset plus Bash-event, plan-revision, and findings-ledger baselines. Dimension selection scans unique paths appended after that boundary, so re-editing a path counts while prior-objective activity does not leak in—even when events share an epoch second. Legacy/resumed sessions without the markers retain a conservative compatibility path.
    - **Baseline dimensions**: generic code quality and prose follow their changed surfaces even for a one-file objective. UI-shaped paths are necessary but not sufficient for design coverage on a new objective: UI/visual semantics, explicit broad scope, a current complex plan, or an active Phase 8 wave must also make design judgment material. `dimension_gate_file_count` is not a suppression threshold for generic code/prose or a reason to force design review.
    - **Required dimensions** (computed from changed surfaces and semantic scope):
        - Any code files → `bug_hunt`, `code_quality`
        - UI-shaped files plus material design semantics/breadth → add `design_quality`. Recognized surfaces include web component/style/markup files; storyboards/XIBs; high-confidence Swift view/screen/controller/cell/row names or `Views`/`Screens`/`UI`/`Presentation` directories; Android `res/layout` XML; and high-confidence Kotlin/Compose UI names/directories. Test/spec paths and generated/minified CSS are excluded.
        - Any doc files, including source-form `.tex`, `.bib`, `.typ`, `.qmd`, and `.rmd` → add `prose`
        - Broad/open-mandate, current complex-plan, or unknown-Bash work → add `completeness` semantically; otherwise cross-surface work adds it only at the configured `dimension_gate_file_count` / `excellence_file_count` breadth
        - At least `traceability_file_count`+ current-objective files (default 6) **and** cross-surface, current-complex-plan, or active-wave scope → add `traceability`
        - Unknown Bash scope → conservatively require generic code quality plus completeness. It adds prose/design only when the prompt's route-time semantics explicitly identify that surface; it never fabricates exact paths or traceability breadth.
    - **Dimension ownership** (one reviewer per dimension; see `AGENTS.md` §"Dimension mapping"):
        - `quality-reviewer` → `bug_hunt`, `code_quality`
        - `excellence-reviewer` → `completeness`
        - `design-reviewer` → `design_quality`
        - `editor-critic` → `prose`
        - `briefing-analyst` → `traceability`
        - `metis` still owns the legacy `stress_test` tick when explicitly dispatched, but stress-testing is plan-phase coverage and is not selected from post-edit file counts
    - **Validity**: each tick records both `dim_<name>_ts` and `dim_<name>_revision`. Revisions are authoritative for new state: code uses the code edit revision, design uses the UI edit revision, prose uses the latest relevant doc/unknown-Bash revision, completeness/traceability use the aggregate edit revision, and stress-test uses the plan revision. Timestamps are migration fallback only. A relevant post-review change invalidates that dimension even inside one second; an unrelated edit cannot erase or refresh its verdict.
    - **Resumed sessions**: sessions resumed from pre-dimension-gate state get one free stop (`dimension_resume_grace_used`) before the gate starts enforcing.
    - **Cap**: 3 blocks (`dimension_guard_blocks`). Exhaustion records `dimensions_missing=...` in `guard_exhausted_detail` and falls through to the excellence gate rather than bypassing all checks.
5. **Completeness/excellence gate**: After all earlier gates pass, broad, cross-surface, open-mandate, unknown-scope, current-complex-plan, or active Phase 8 wave-plan work at the configured current-objective breadth requires a fresh `excellence-reviewer`; an arbitrary cumulative file count alone does not. Cap: 1 block per edit generation (controlled by `excellence_guard_triggered` plus its revision companion).
6. **Metis-on-plan gate** (opt-in): When `OMC_METIS_ON_PLAN_GATE=on` and the recorded plan is flagged high-complexity, blocks the stop until `metis` has stress-tested the current plan since `plan_ts`. Cap: 1 block per plan cycle (`metis_gate_blocks`).
7. **Closeout-seal gate**: After work/evidence gates are clean, material execution can release only when `closeout-preflight.sh` sealed the exact current work/evidence fingerprint. The isolated probe skips this one recursive check; the live Stop never does. A missing or stale seal continues work without exposing the full internal gate transcript.
8. **Final-closure gate**: After the work itself is clean and the seal is current, checks whether Claude's final user-facing wrap is audit-ready. Material sessions require standalone `Changed`/`Shipped`, `Verification`, `Objective coverage` (or `/goal`'s stricter `Goal achieved`), and `Next` labels, plus `Risks` when anything was deferred. The response must name the exact recorded verification command and every bounded required literal anchor derived from current-cycle paths, deferred IDs, and strong facts in a prior candidate. There is no arbitrary length floor: a detailed report may put headings near the top, while anchors and whole-objective coverage prevent a thin “fixed the last gate” delta from stranding facts.

The preflight seal is a readiness certificate, not permission to skip live checks: Stop always re-runs the complete guard against real state. Clean execution, advisory, and no-edit releases finish with an optimistic generation CAS. In addition to edit/code/doc/plan clocks, a completed closeout rechecks the full readiness fingerprint under the release lock, covering reviewer, verification, scope, delivery, objective, material-tool generation, and bounded artifact signatures. If any relevant work/evidence changes while gates run, Stop emits a retry and leaves the session active. Revisionless migrated state retains timestamp fallbacks; presentation/timing/counter keys are excluded.

Gate-specific caps retain their configured block/scorecard semantics. Separately, the dispatcher counts every Stop continuation and emits an explicit exhausted receipt one attempt before the configured platform ceiling (`max(1, cap-1)`; seven under the default cap of eight); no hidden candidate is allowed to disappear at that boundary.

**`skills/autowork/scripts/mark-edit.sh`** -- **PostToolUse hook** for Edit, Write, MultiEdit, and NotebookEdit, plus the Bash edit-clock handler invoked by `posttool-dispatch.sh` (and directly on `PostToolUseFailure:Bash`). Records `last_edit_ts` on every observed edit (backward compat) and classifies exact paths via `is_doc_path` and `is_ui_path`: code/notebook edits bump `last_code_edit_ts`; source docs (including `.tex`, `.bib`, `.typ`, `.qmd`, `.rmd`) bump `last_doc_edit_ts`; recognized web/native UI paths also bump the UI-specific clock/revision used by design freshness. Bash has no authoritative file path, so `pretool-intent-guard.sh` records tool-use-id-keyed before state for every non-trivially-read-only foreground Bash call in a Git worktree. Ordinary candidates use a fast NUL-safe fingerprint over HEAD, porcelain-v2 index/worktree records, and batched no-filter content IDs for dirty tracked **and untracked** files; materialized skip-worktree/assume-unchanged paths are added explicitly because porcelain hides them. Snapshot plumbing uses a trusted system PATH (FHS/Homebrew plus immutable/system and per-user Nix profiles from the caller) and disables fsmonitor, external diff, pagers, optional locks, and hooks so observation cannot execute arbitrary caller-shadowed binaries or ambient Git helpers. Git/gh inspection is never placed on the proven-read-only skip because the user's actual command may execute a configured helper. Delivery (`git add`/commit/push/tag and `gh` publication) uses a normalized temporary-index worktree tree: it stays stable across expected index/HEAD transitions but changes when pre-commit/pre-push hooks rewrite or stage worktree bytes. Dirty snapshots over 256 paths or 64 MiB, unmerged indexes, and dirty submodules/nested worktrees fail closed instead of hashing without bound or sampling attacker-chosen paths. Recognized write syntax is the fallback for ignored content, missing snapshots, non-Git/ambiguous roots, failed-after-write calls, argument-aware Python positional/keyword write modes, and literal executor bodies (`sh -c`, `eval`, and data-driven `xargs`); every asynchronous candidate fails closed at launch because an immediate after-state is unsound. Intent and delivery parsing share a NUL-framed, quote-aware compound segmenter and executable-position launcher grammar. A quote/depth-aware nested-executor walker classifies executable `$()`/backticks/process substitutions and literal `sh -c`/combined-option shell, `eval`, and `env -S` bodies through common wrappers, normalizes Git top-level flags inside each body, and preserves commit-vs-publish kind separation for explicit forbidden contracts; opaque split strings fail closed. Ordinary literal command-like prose remains data. Quoted heredoc bodies and inputs beyond the four-level / 4096-byte / 16-opener parser budget are conservatively denied at intent time, but never count as delivery evidence. Delivery evidence masks nested executables before segment parsing and records only a status-proven direct outer action or `&&` chain, because an outer success cannot prove a nested action succeeded. Git flag semantics remain verb-specific (`push -n` is dry-run; `commit -n` is mutating `--no-verify`). A proven simple out-of-worktree scratch/output target is suppressed. Exact-path tools alone maintain code/doc/UI counts; unknown Bash scope sets `bash_unknown_edit_scope=1` plus `last_bash_edit_ts` without inventing a file count. Gate selection treats that uncertainty conservatively as generic code plus completeness coverage and promotes the derived session risk to high; route-time prose/UI semantics may add those reviewers, but no exact surface is fabricated. Path-derived inferred-contract rules cannot infer a specific adjacent surface without an exact path, so reviewers carry that residual. Remaining observation limits are opaque ignored-file-only writes with no recognized syntax (including dynamic/ANSI-C-expanded executor bodies), opaque non-Git/outside-root mutations, and delivery hooks that write only ignored/Git-clean-equivalent bytes. Resets `stop_guard_blocks`, `session_handoff_blocks`, `advisory_guard_blocks`, and `stall_counter`; excludes internal Claude paths for exact-path tools; and logs exact paths to `edited_files.log`.

Connector-managed binary artifacts are a separate residual. A `.docx`, `.pptx`, or `.xlsx` write performed through a connector/MCP tool may not expose an exact path to the ordinary editor hooks, so its existence alone is not hook-level edit/review evidence. Native artifact workflows must verify the saved file through their connector-specific render/recalculation/inspection loop. Likewise, recognizing a source document extension routes prose review but does not prove TeX/Typst/Quarto/R Markdown compilation; use the project's real build/render command when applicable.

**`skills/autowork/scripts/record-tool-start-revision.sh`** -- **PreToolUse hook** for Bash and MCP calls. Writes a tool-use-ID-keyed `.verification-starts/` sidecar containing the current `last_code_edit_revision` before the tool runs. This evidence path is deliberately independent of optional timing telemetry. Calls without a tool-use ID cannot be paired safely, so their later verification result fails closed. Denied/failed/no-completion calls may leave an abandoned sidecar; a later start best-effort sweeps entries older than one day, `/ulw-off` clears the directory as an active-interval boundary, and normal session retention is the final cleanup boundary. Those leftovers cannot be replayed as accepted evidence because result consumption validates both tool ID and tool name and removes the sidecar under the state lock.

**`skills/autowork/scripts/record-verification.sh`** -- **PostToolUse hook** for Bash and MCP verification tools (Playwright browser_snapshot/take_screenshot/console_messages/network_requests/evaluate, computer-use screenshot). For Bash tools, checks if the command matches a test/build/lint pattern (npm test, cargo test, pytest, vitest, eslint, tsc, etc.). If mark-edit recorded the same Bash tool call as edit-bearing, a transient marker makes verification reject that call entirely; a separate post-edit verification invocation is required, so `tests; mutate` cannot self-verify. For MCP tools, classifies the tool name via `classify_mcp_verification_tool` and detects pass/fail via `detect_mcp_verification_outcome`. Under one state lock, it consumes the matching PreToolUse snapshot, compares its start revision with the current code revision, and only then commits `last_verify_*` state. An edit that lands while a test/browser check is in flight makes that result stale; missing, malformed, mismatched, or replayed snapshots fail closed and leave prior accepted verification evidence untouched while writing `last_stale_verify_*` diagnostics. MCP verification base scores are below the default confidence threshold (40) — passive observations only pass the gate when output carries assertion/pass-fail signals or when recent edits include UI files (which adds a context bonus).

**`skills/autowork/scripts/record-delivery-action.sh`** -- **PostToolUse hook** for Bash delivery actions. Records successful `git commit` as commit actions and successful `git push`, `git tag`, and `gh pr/release/issue` publish-class commands as publish actions. Dry-run/read-only forms (`git commit --dry-run`, `git push --dry-run`, read-only `git tag`, `git apply --check|--stat|--numstat|--summary`) do not satisfy the contract. Feeds `stop-guard.sh` so prompt-stated commit and publish requirements are enforced by observed actions, not final-answer prose.

**`skills/autowork/scripts/record-advisory-verification.sh`** -- **PostToolUse hook** for Grep and Read. During advisory tasks, records `last_advisory_verify_ts` when a non-internal file is read or searched. Also implements stall detection: increments a counter on each Read/Grep call, and at `stall_threshold` consecutive calls (default 12, configurable via `oh-my-claude.conf`) without an edit, test, or agent delegation, injects a stall-check nudge.

**`skills/autowork/scripts/reflect-after-agent.sh`** -- **PostToolUse hook** for Agent. After an agent returns, injects only a structured capsule (`agent`, universal `VERDICT`, structured finding count, and up to five stable finding IDs) plus a short verification/wait instruction. It deliberately does not copy agent prose already delivered by the native Agent result, avoiding duplicate context and prompt-injection amplification. For advisory tasks, it additionally warns not to synthesize until all required returns arrive.

**`skills/autowork/scripts/record-reviewer.sh`** -- **SubagentStop hook** for reviewer and analysis agents (`quality-reviewer`, `editor-critic`, `excellence-reviewer`, `metis`, `briefing-analyst`, `design-reviewer`, plus `superpowers:code-reviewer` / `feature-dev:code-reviewer`). Accepts a reviewer-type argument (`standard|excellence|prose|stress_test|traceability|design_quality`) that determines which dimension and clock it owns. Only an exact reviewer `VERDICT: CLEAN|SHIP|FINDINGS (N)|BLOCK (N)` on the final non-empty line is authoritative; a malformed or non-final `VERDICT:` fails closed, while the legacy phrase detector is reserved for pre-contract output with no verdict token at all. Only the standard code-review path records `last_review_ts`, `review_had_findings`, and generic format state. Prose records `last_doc_review_ts` plus isolated `doc_review_had_findings`; other reviewer roles update only their own state. Current dispatch records carry a version, the monotonic objective cycle, the platform-issued native `agent_id`, and the canonical dimension revision (including design's UI-to-code fallback); after the native/session markers are armed, missing, malformed, uncommitted, replayed, wrong-cycle, or generation-mismatched starts fail closed without touching accepted reviewer evidence. In-flight sessions created before `SubagentStart` binding retain an explicit migration path. Accepted reviewer metadata/dimension state commits in one locked batch and append a bounded objective/revision-scoped row to `review_history.jsonl`; structured findings there are evidence only and never tick a dimension. `quality-reviewer` is currently the sole role with an explicit remediation mode that consumes its same-role anchors before widening when the fix crosses contracts or surfaces. Other roles, including prose-native `editor-critic`, keep their normal review contracts. The PreTool dispatch gate keeps one live call per exact identity so the subsequent native bind is unique; distinct reviewer identities still run in parallel. A fresh `[review-rebind:<id>]` explicitly authorizes replacement of a confirmed killed call, while native identity—not model-authored text—selects the completing row.

**Reviewer batch stability.** `stop-guard.sh` reports the complete minimal missing reviewer set in one block so distinct roles can run concurrently against a settled revision; when quality and completeness are both required it assigns `quality-reviewer` defects-only mode and leaves completeness/polish to `excellence-reviewer`. Council Phase 8 additionally prefixes every selected reviewer and semantic-risk specialist description with `[review-batch]`. While Phase 8 is active, `record-pending-agent.sh` converts that exact per-dispatch marker into a deterministic ID derived from the objective timestamp and settled edit revision; it is compatible with a preceding `[council:*]` marker and does not rely on a permanent role allowlist. While any current generation-stamped reviewer or marked batch row remains pending, `pretool-intent-guard.sh` denies recognized workspace mutation but permits inspection/verification, preventing one edit from invalidating several paid reviews. Pending rows older than two hours no longer freeze edits, so a killed Agent call cannot trap the user indefinitely. After all returns, the main thread reconciles once, remediates once, and re-runs only affected roles. **`quality-pack/memory/core.md`** -- Loaded into every session via CLAUDE.md. Defines thinking quality standards (reason before acting, diagnose before fixing, track progress), workflow rules (classify intent, classify domain, route to specialists, test after changes, review before stopping), code quality rules (no placeholder stubs, no restating comments), anti-patterns (forbidden behaviors like asking "should I proceed?"), and failure recovery (stop retrying after 3 failures, switch approach or delegate to oracle).

## Compaction Continuity

When Claude Code compacts a session — either automatically when the context budget is approaching the limit, or manually via `/compact` — the harness runs three hooks to preserve working state:

1. **`PreCompact`** → `pre-compact-snapshot.sh`: writes a critical-first `precompact_snapshot.md` priority manifest with bounded objective/contract, review/verify state, remaining obligations, pending specialists, edited files, plan, and capped narrative hints, plus a `review_pending_at_compact` flag. Back-to-back compactions archive an unconsumed prior snapshot as `precompact_snapshot.<mtime>.md` (retained up to 5 files).

2. **`PostCompact`** → `post-compact-summary.sh`: reads Claude Code's `.compact_summary` field (or emits a fallback when empty), retains a bounded diagnostic copy without duplicating the runtime-supplied summary into context, writes the priority manifest to `compact_handoff.md`, and sets a `just_compacted` flag so the next `UserPromptSubmit` can bias classification toward continuation. `HOOK_DEBUG=true` dumps the raw hook JSON to `compact_debug.log` for schema diagnosis.

3. **`SessionStart` (source=compact)** → `session-start-compact-handoff.sh`: injects the bounded priority manifest via `additionalContext`, re-asserts ultrawork or the prior non-execution intent, lists interrupted specialist identities, enforces pending-review requirements, and forces one full router refresh before edge-triggered deltas resume.

Pending specialist tracking uses `record-pending-agent.sh` on both PreToolUse Agent and SubagentStart. Admission permits at most 64 live calls and one live row per exact identity; ledger rotation preserves every live/malformed row and removes only redundant tombstones beyond the newest 32. SubagentStart binds the platform `agent_id`, with `native_agent_bindings.jsonl` as the last-write commit marker. Fresh execution atomically increments `review_cycle_id`, permanently taints every older live identity for the session, and converts those rows to hidden abandoned tombstones; a true continuation preserves the cycle and live rows. The taint and accepted-authorization registries survive tombstone rotation. Same-identity reuse after a confirmed killed, expired-claim, prior-objective, malformed-ended, or deactivated call requires `[review-rebind:<fresh-id>]`; native clients bind the new call automatically, while `REVIEW_DISPATCH_ID` is the pre-native compatibility path. At SubagentStop, summary claims add `completion_claim_id`, `completion_claim_ts`, and `completion_claim_effects_complete`; a fresh claim denies replacement, an incomplete claim owner has a 120-second recoverable lease, and a summary-first reviewer/planner claim remains effects-complete only until its causal start is consumed. Completion consumers require the committed native ID and current cycle before interpreting content, so either hook order is safe and an abandoned/missing-ID/prior-objective result cannot write summaries, agent-first evidence, design state, discovered scope, uncertainty evidence, dimensions, or Council returns. Dispatch admission mutates its coordinated artifacts transactionally; an interrupted ready journal fails closed until `/ulw-off` clears the interval rather than restoring an old snapshot over newer work. The two-hour row TTL independently releases mutation freeze for a killed call; it is not a prerequisite for explicit recovery.

On `/ulw-off`, `ulw-deactivate.sh` first taints every live exact dispatch identity, then clears `pending_agents.jsonl`, `agent_dispatch_starts.jsonl`, interrupted dispatch/native-bind/plan transaction journals, and `.verification-starts/` under the session lock. It deliberately retains the taint/rebind/native-binding registries and reviewer/planner/all-agent/native tracking versions, so reactivation requires an explicit fresh binding for a reused identity and any late pre-deactivation completion fails closed.

## Intent Classification Order

The classification order in `classify_task_intent()` is a protected design decision. Changing it affects routing accuracy across all prompts. The order:

1. **Continuation** (`is_continuation_request`) -- "continue", "resume", "carry on", "keep going", "pick it back up". Matched first because continuation prompts often contain advisory-like words ("continue where you left off?") that would otherwise misclassify.

2. **Checkpoint** (`is_checkpoint_request`) -- "checkpoint", "pause here", "stop here", "for now", "wave 1 only", "first phase only". Explicit user requests for phased delivery.

3. **Session management** (`is_session_management_request`) -- Questions about session strategy: "should I start a new session?", "is the context budget okay?". Requires both session-related keywords AND question-form syntax.

4. **Imperative** (`is_imperative_request`) -- Polite commands: "can you fix...", "please implement...", "go ahead and...", "I need you to...". Checked BEFORE advisory because "can you fix the bug?" is an action request, not advice-seeking. The verb list covers ~40 common action verbs. Two additional branches catch natural-English imperative shapes the head-anchored patterns miss: **tail-position imperative** (a prompt that opens advisory but closes with a destructive verb after a sentence boundary — `Review the plan. Then commit the changes.`) and **implementation-verb-led conjunction** (v1.23.0; `Implement and then commit as needed`, `Build it, then push to origin`, `Refactor X and tag v2.0` — impl-verb head + (and|,) conjunction + destructive verb + object-marker tail).

5. **Advisory** (`is_advisory_request`) -- Questions and opinion-seeking: "should we...", "what do you think...", "is it better to...", "pros and cons". Only matched if none of the above patterns triggered first.

6. **Default: execution** -- If nothing else matches, the prompt is treated as a direct execution request.

**Defense-in-depth: prompt-text trust override (v1.23.0)** -- If the classifier mis-routes despite the widened patterns, `pretool-intent-guard.sh` re-reads `recent_prompts.jsonl` and re-runs `is_imperative_request` against the most recent user prompt before blocking a destructive op. When EVERY destructive non-allowed segment in the attempted command has its verb authorized in the prompt with an imperative-tail object marker (or at end-of-prompt), the guard allows and records `gate=pretool-intent` `event=prompt_text_override` for audit. Compound-command safety: `git commit && git push --force` only passes when both `commit` and `push` are authorized in the prompt — not just the first one. Conf-gated by `prompt_text_override` (default `on`).

**Bias-defense directive layer (router-side)** -- The prompt-intent-router emits up to three directives on fresh-execution prompts to defend against classifier-blind biases (the classifier got the intent right but the *prompt shape* hides a bias):

- `prometheus_suggest` (default off; declare-and-proceed since v1.24.0) — defends against over-commitment when an execution prompt is short, product-shaped, and unanchored. The directive tells the model to state its scope interpretation in one or two declarative sentences as part of its opener and proceed; the user can redirect in real time. The flag does NOT instruct a hold.
- `intent_verify_directive` (default off, suppressed when prometheus already fired; declare-and-proceed since v1.24.0) — defends against over-commitment when an execution prompt is short and unanchored. The directive tells the model to state its interpretation of the goal in one declarative sentence as part of its opener and start work; the user can redirect in real time. Pause case is narrow: only when confidence is low AND the wrong call would be hard to reverse. The flag does NOT instruct a hold.
- `exemplifying_directive` (default on, v1.23.0) — defends against under-commitment when the prompt uses example markers (`for instance`, `e.g.`, `such as`, etc.) by instructing the model to treat the example as one item from an enumerable class. Symmetric to (but opposite of) the other two; fires INDEPENDENTLY because narrowing and widening are orthogonal.
- `exemplifying_scope_gate` (default on) — hardens `exemplifying_directive` by requiring `exemplifying_scope.json` before Stop. The model records sibling items via `record-scope-checklist.sh init`; each item must become `shipped` or `declined` with a concrete WHY. This closes the gap where the directive could be ignored and the literal example still shipped alone.

The three directives emit `gate=bias-defense` `event=directive_fired` rows for `/ulw-report` audit. The hard exemplifying-scope gate emits `gate=exemplifying-scope` block/exhausted rows. The mark-deferred validator emits `gate=mark-deferred` `event=strict-bypass` rows when `OMC_MARK_DEFERRED_STRICT=off` lets a would-be-rejected reason slip through (audit row carries the truncated reason under `.details.reason`); `/ulw-report` aggregates these into the "Mark-deferred strict-bypasses" section so silent-skip patterns remain visible across sessions.

## Domain Scoring

`infer_domain()` scores the prompt text against four keyword lists using case-insensitive grep:

- **Coding**: ~35 keywords (bug, fix, refactor, implement, component, endpoint, API, schema, deploy, docker, etc.)
- **Writing**: ~20 keywords (draft, essay, article, report, proposal, email, manuscript, cover letter, etc.)
- **Research**: ~20 keywords (research, investigate, analyze, compare, benchmark, audit, evaluate, etc.)
- **Operations**: ~15 keywords (plan, roadmap, timeline, agenda, meeting, follow-up, checklist, prioritize, etc.)

Each domain gets a score equal to the number of keyword matches. The domain with the highest score wins. If all scores are zero, the domain defaults to `general`.

**Mixed domain**: Triggered when coding has a nonzero score and a second domain scores at least 40% of the primary domain's score. This captures prompts like "research the API options and implement the best one" where both research and coding are significant.

## Paradigm choices and alternatives considered

This section is the "why this shape, not another" companion to the "what" above. Read it before contemplating a paradigm shift — the alternatives were enumerated in v1.32.x via a `divergent-framer` pass (the post-v1.31.3 advisory item 10, deferred until items 1-9 stabilized) and the conclusion was **no shift**, with explicit redirect-if conditions named below.

### The current paradigm (Framing 1 — chosen)

**Prompt-injected directive substrate.** The harness is transparent middleware: `prompt-intent-router.sh` classifies the user's prompt and reshapes the model's context window with text directives (`EXEMPLIFYING SCOPE DETECTED`, `INTENT-BROADENING DIRECTIVE`, `DIVERGENT-FRAMING DIRECTIVE`, `EXHAUSTIVE AUTHORIZATION DETECTED`, etc.). `closeout-preflight.sh` privately certifies a stable generation before final prose, and the single `stop-dispatch.sh` owner invokes `stop-guard.sh` to enforce completion with tiered caps (3 review/verification, 2 session-handoff and discovered-scope, 1 advisory and excellence). State is per-session JSON; cross-session aggregates are JSONL. Specialists are `.md` agent files with `disallowedTools` boundaries. Skills are thin `SKILL.md` wrappers.

**Why it wins for the current scope.** The constraints — solo-dev cognitive overhead, substrate stability (bash + jq is preinstalled everywhere), Anthropic relationship (the harness rides published Claude Code hook surfaces, never forks), forward velocity (16 v1.32.x releases in one week of autonomous-loop sessions), adversarial robustness (recent 4-attacker security review closed via local-file primitives + `disallowedTools`) — all favor the status quo. The trade-off the paradigm makes is sophistication for installability; that trade is correct as long as the project stays solo-dev or single-team and Claude-Code-only.

### Alternatives considered and rejected (for now)

| Framing | Mental model | Why rejected for current scope |
|---|---|---|
| **MCP-first capability substrate** | Quality gates / classifiers / memory exposed as typed MCP tools the model calls explicitly. | Install friction (Node/Python server with auth + transport + lifecycle); Claude-Code-side gate semantics become "the model has to *choose* to call them" — exactly the loop the prompt-injection paradigm was built to remove. |
| **Output-style + slash-command shell** | Native Claude Code config bundle only. No hooks, no state, no classifier. | Cannot enforce — only suggest. The 3-block stop cap, FINDINGS_JSON parsing, and per-session counters that drive measurable outcomes do not exist. The harness's differentiator is enforcement; this paradigm gives that up. |
| **Telemetry-first, intervention-second** | Observability primary; directives derived from measured outcomes. | Telemetry surface is itself an attack target; feedback loop is statistically slow (n=many sessions per directive); without baseline-comparison framework, telemetry is bookkeeping cost without decision yield (see the v1.32.4 router cost audit Phase 2-4 deferrals — the canonical evidence). |
| **Sub-agent orchestrator (graph-of-roles)** | Main thread navigates a state machine; every prompt routes through a planner→researcher→implementer→reviewer graph. | **Split verdict (updated).** Rejected *as the default paradigm* — latency/cost balloon (every node is a model call) and it breaks the "ULW = momentum" expectation on small tasks; both still hold. But the third original reason — "graph state needs a workflow engine we don't have" — **expired** when Claude Code shipped its `Workflow` tool. So the engine is now **embraced as an opt-in substrate** for heavy fan-out (`workflow_substrate=on`; council Phase 8 waves, large audits/migrations) while the default per-prompt path stays prompt-injected. *Substrate, not paradigm* — see `model-robustness.md` Mechanisms 2–3 and the `workflow_substrate` flag. A documented rejection is not permanent truth (`core.md`): when a premise lifts, re-examine. |

### Redirect-if conditions

The current paradigm is correct *for the current scope*; it is not correct *forever*. Revisit the rank when **either** of these holds:

- **Distribution shift.** The harness ships to ≥10 external users. Install-friction cost is now amortized across users instead of paid by one solo dev; typed contracts (MCP) become non-negotiable for support overhead reasons.
- **Surface shift.** The harness needs to run inside non-Claude-Code IDEs (Cursor, Windsurf, Zed). Cross-IDE portability is non-negotiable; MCP becomes the only credible substrate.

When either trips, dispatch `divergent-framer` again with the updated constraints and re-rank. **Framing 2 (MCP-first) is the most likely successor.**

### Within-paradigm improvement vectors (surfaced by the framer's "Makes hard" rows)

These are not paradigm shifts but real costs of the current paradigm worth chipping at as polish work:

- **Directive wording drift.** Behavior is text-coupled; tiny edits to directive text silently change outcomes. The router cost audit (Phase 1 shipped in v1.32.x) was the first instrumentation step; Phases 2-4 (token analysis, baseline comparison, firing-rate × cost cutoff) remain deferred. Worth resuming when classifier_telemetry data accrues enough to make the deltas statistically distinguishable.
- **Conf-flag interactions.** The bias-defense flags can fight each other across long sessions (the test-isolation conf-leak case is the canonical symptom). A composability matrix in `docs/customization.md` would be the cheap win.
- **FINDINGS_JSON brittleness across model versions.** Currently regression-netted by `tests/test-findings-json.sh` against the contract shape. Worth re-running the regression set against new Claude model releases (Sonnet/Opus/Haiku rev bumps) before assuming the JSON shape will hold.
- **Classifier accuracy without formal eval.** `tools/classifier-fixtures/regression.jsonl` exists; a precision/recall surface against a labeled corpus would let directive tuning be evidence-driven rather than vibes-driven.

These are recorded here (not as open issues) because they're polish work for an already-stable paradigm, not blockers for a shift to a different one.
